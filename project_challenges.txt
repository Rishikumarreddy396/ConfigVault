Project Challenges and Solutions

Architectural and Database Modeling Challenges

Issue: When setting up the relationship between Config and ConfigVersion, attempting to return raw entities directly from the controller caused infinite recursion during JSON serialization.
Solution Approach: I implemented Data Transfer Objects (DTOs) to decouple the database entities from the API responses. This prevented Jackson from falling into infinite loops and kept the API contract clean and predictable.

Issue: Exception handling was initially inconsistent. Database constraint violations were throwing generic 500 Internal Server Errors, exposing stack traces to the client and making it hard to debug client-side.
Solution Approach: I created a GlobalExceptionHandler using @ControllerAdvice. By catching specific exceptions like ResourceNotFoundException and MethodArgumentNotValidException, I was able to return a standardized JSON error shape with appropriate HTTP status codes like 404 and 400.

Business Logic and Versioning Challenges

Issue: Generating accurate line-by-line diffs for the version history was complex to write from scratch. It was highly prone to edge cases, especially regarding different operating system line endings causing false positive diffs.
Solution Approach: Rather than reinventing the wheel, I integrated the java-diff-utils library to handle the heavy lifting of generating unified diff patches. I ensured that string inputs were normalized to uniform line endings before passing them into the diff generator.

Issue: Promoting a configuration to a new environment required copying data while maintaining a clear history of where the configuration originally came from.
Solution Approach: I designed the promotion logic to clone the latest content into a brand new Config entity for the target environment, rather than mutating the original. Explicit audit trail entries were added to both the source and target configurations to establish a clear lineage.

Security and Authentication Hurdles

Issue: Upgrading to Spring Security 6 introduced major deprecations. The traditional WebSecurityConfigurerAdapter was removed, which broke many older approaches to setting up JWT authentication.
Solution Approach: I migrated to the modern component-based security configuration by defining a SecurityFilterChain bean. I then created a custom JwtAuthenticationFilter extending OncePerRequestFilter to manually extract, validate, and inject the authenticated user into the SecurityContextHolder.

Issue: After securing the application, Swagger UI was completely blocked. Even after whitelisting the paths, it lacked a way to pass the JWT token to the backend, rendering secured endpoints untestable via the UI.
Solution Approach: I explicitly whitelisted the swagger-ui and api-docs paths in the SecurityFilterChain. Then, I created an OpenApiConfig bean to programmatically declare a Bearer Auth security scheme, which automatically added an Authorize button to the Swagger interface.

Infrastructure and Deployment Issues

Issue: When setting up docker-compose, the Spring Boot container was starting up faster than the PostgreSQL container could fully initialize. This caused the application to throw connection refused errors and crash immediately on startup.
Solution Approach: I configured a healthcheck for the postgres service in the docker-compose.yml file. I updated the app service to use the depends_on block with the condition: service_healthy, forcing the application to wait until the database was fully ready to accept connections.
